## 项目概述

TravenManager 是一个为 SillyTavern 设计的配置管理工具，提供精美的 GUI 界面，帮助用户快速便捷地管理 SillyTavern 的各项配置并自动优化其中的数据（如世界书和角色卡）。

### 主要功能

- 一键启动 SillyTavern
- 实时查看 SillyTavern 启动日志
- 管理 SillyTavern 配置
- 检查 SillyTavern 更新
- 优化 SillyTavern 数据
- 世界书管理与优化
- 角色卡管理与优化
- 标签管理系统
- 记忆插件系统
  - Excel模板管理
  - 聊天记录管理
  - 表格关联功能
  - 表格整理功能

## 项目架构

TravenManager 采用 Electron + React + TypeScript 的架构，分为主进程和渲染进程两部分：

### 主进程

- 负责与系统交互，管理 SillyTavern 进程
- 提供 IPC 接口，与渲染进程通信
- 处理文件系统操作和网络请求

### 渲染进程

- 负责用户界面渲染
- 处理用户交互
- 通过 IPC 与主进程通信

### 数据流

1. 用户在渲染进程中操作界面
2. 渲染进程通过 IPC 向主进程发送请求
3. 主进程处理请求，执行相应操作（如启动 SillyTavern）
4. 主进程将结果通过 IPC 或 WebSocket 发送回渲染进程
5. 渲染进程更新界面显示

### 数据流设计

记忆插件系统的完整数据流：

1. **用户操作**：
   - 在模板管理界面创建/编辑 Excel 模板
   - 在聊天记录管理界面浏览聊天记录
   - 选择聊天记录并关联模板
   - 触发表格整理功能
2. **渲染进程处理**：
   - ChatManager/TemplateManager 组件接收用户输入
   - 通过 Zustand 管理组件状态
   - 调用 window\.electronAPI.memory.\* 方法
3. **IPC 通信**：
   - 通过 preload.ts 暴露的 IPC 接口
   - memoryHandlers.ts 处理渲染进程请求
   - 主进程日志通过 sendLogToRenderer 发送回渲染进程
4. **主进程服务**：
   - chatLogService.ts 处理聊天记录管理
   - excelTemplateService.ts 处理 Excel 文件操作
   - 配置加载优先使用用户自定义配置
5. **AI 交互**：
   - 调用配置的 AI 模型 API
   - 使用优化的提示词引导 AI 返回结构化数据
   - 解析 AI 返回的 JSON 响应
6. **数据持久化**：
   - 安全文件名处理（移除非法字符）
   - Excel 文件通过 fs + XLSX 库读写
   - 模板副本与原始模板分离存储

### 13. AI 请求处理机制

#### 功能定位与业务价值

AI 请求处理机制是 TravenManager 的核心技术创新，通过在主进程中处理 AI 请求，避免了前端直接发送请求导致的 CORS 问题，确保了 AI 功能的稳定运行。该机制支持多种 AI 引擎配置，包括不同的 API 模式和密钥传输方式。

### 19. 外部系统调用 API

#### 核心功能特性

- 适用场景：实时对话过程中需要立即整理当前对话内容

## 项目开发过程中遇到的问题及解决方案

### 1. 启动流程异常（白屏问题）

**问题**：当 TravenManager 启动后，点击启动按钮，SillyTavern 能够正常启动，但 TravenManager 界面会出现白屏现象，所有 UI 元素均不可见且无法交互。

**根本原因**：Vite 开发服务器在 SillyTavern 启动后崩溃，导致渲染进程失去了与开发服务器的连接。

**解决方案**：

- 在生产模式下运行应用程序，避免使用 Vite 开发服务器
- 优化 Vite 配置，使用不同的端口，避免与 SillyTavern 或其他服务冲突

### 2. 进程管理问题

**问题**：当点击 TravenManager 窗口右上角的关闭按钮(X)时，已启动的 SillyTavern 进程未能正常终止，导致其持续占用 8003 端口资源。

**根本原因**：进程终止逻辑不够完善，没有处理进程退出超时的情况。

**解决方案**：

- 添加了 `forceCleanup` 方法，用于执行最高级别的清理，强制终止所有 SillyTavern 进程
- 添加了进程终止的超时处理，确保进程能够被正确终止
- 在窗口关闭事件处理中添加了 0.5 秒的延时，确保所有 SillyTavern 进程被清理
- 实现了跨平台兼容的进程清理机制，支持 Windows 和 Unix-like 系统

### 3. 端口冲突与二次启动失败

**问题**：当再次启动 TravenManager 并点击启动按钮时，系统会提示 8003 端口已被占用（此问题由上一轮未正常停止的 SillyTavern 进程导致），同时 TravenManager 再次出现白屏问题。

**根本原因**：SillyTavern 进程没有被正确终止，端口不会被释放，导致再次启动时出现端口冲突。

**解决方案**：

- 优化了进程终止逻辑，确保 SillyTavern 进程能够被正确终止
- 添加了进程退出事件监听器，确保进程状态能够被及时更新
- 添加了超时处理，确保即使进程没有正常退出，也能释放进程资源
- 实现了 `killProcessByPort` 方法，用于终止占用指定端口的进程

### 4. 中文日志乱码问题

**问题**：在控制台输出的中文日志出现乱码。

**根本原因**：控制台编码与输出编码不匹配。

**解决方案**：

- 将所有的中文日志改为英文，避免乱码现象
- 确保控制台输出的日志能够正确显示

### 5. 日志处理优化

**问题**：SillyTavern 启动时产生大量日志，导致渲染进程因频繁更新而崩溃。

**根本原因**：日志处理没有优化，导致渲染进程频繁更新状态。

**解决方案**：

- 添加了日志缓冲区和批量发送功能，避免渲染进程因频繁更新而崩溃
- 限制了日志长度，避免发送过大的日志
- 批量处理日志，每 100ms 发送一次，减少 IPC 通信的频率

### 6. WebSocket 连接优化

**问题**：WebSocket 连接不稳定，导致日志传输中断。

**根本原因**：WebSocket 连接没有正确处理错误和重连。

**解决方案**：

- 实现了 WebSocket 长连接，用于传输日志
- 添加了详细的错误处理，确保 WebSocket 连接的稳定性
- 在 WebSocket 连接断开时自动重连

### 7. React 版本不兼容问题

**问题**：前端出现 "Minified React error #321" 错误，导致白屏。

**根本原因**：React 19.2.4 版本与 Ant Design 6.3.5 不兼容，导致 hook 调用异常。

**解决方案**：

- 将 React 和 React DOM 的版本从 19.2.4 降级到 18.2.0，确保与 Ant Design 6.3.5 兼容
- 清理依赖，删除 node\_modules 目录和 package-lock.json 文件
- 重新安装依赖，确保依赖关系的一致性

### 8. 暗色模式显示问题

**问题**：点击右上角的"暗色"按钮后，页面显示异常，顶部栏仍然为白色，日志组件没有正确切换到暗色模式。

**根本原因**：

- Header 组件的暗色模式样式选择器不正确
- 日志组件的内联样式覆盖了 CSS 类中定义的暗色模式样式

**解决方案**：

- 修改 App.tsx 文件，让 Layout 组件的 className 根据主题状态动态变化

### 9. 系统白屏，前端报错

**问题**：系统白屏，前端报错：`Uncaught TypeError: Cannot read properties of undefined (reading 'map')` at Settings.tsx:652:34

**根本原因**：当配置对象不完整或为 null/undefined 时，尝试访问其属性导致了错误。

**解决方案**：

- 修改了 `config?.aiEngines.map()` 为 `(config?.aiEngines ?? []).map()`，确保即使配置为 null 也能返回空数组
- 添加了 `const engines = config.aiEngines || [];`，确保即使配置中没有 aiEngines 字段也能正常处理
- 修改了 `config.aiEngines.map()` 为 `(config.aiEngines || []).map()`，确保即使配置中没有 aiEngines 字段也能正常处理
- 修改了 `[...config.aiEngines]` 为 `[...(config.aiEngines || [])]`，确保即使配置中没有 aiEngines 字段也能正常处理

**预防措施**：

- 使用可选链操作符 (`?.`) 和空值合并操作符 (`??`) 来处理可能为 null 或 undefined 的值
- 在关键位置添加默认值处理，防止在配置不完整时出现错误

### 10. 添加引擎按钮无响应

**问题**：点击"添加新引擎"或"添加引擎"按钮，没有反应，没有弹出类似引擎配置的输入框。

**根本原因**：`handleAddEngine` 函数中设置 `setEditingEngine(null)` 导致模态框无法正确显示添加引擎的表单。

**解决方案**：

- 修改了 `handleAddEngine` 函数，将 `setEditingEngine(null)` 改为 `setEditingEngine({} as AIEngineConfig)`，确保模态框显示添加引擎的表单
- 调整了模态框的逻辑，根据 `editingEngine` 和 `editingEngine.id` 的值来显示不同的标题和按钮

**预防措施**：

- 确保在添加新项时，设置一个空对象作为编辑状态，而不是 null
- 模态框的显示逻辑应该根据编辑状态的类型来动态调整

### 11. API 密钥传输方式兼容问题

**问题**：有些 AI 引擎的 key 是以请求头：`Authorization: Bearer xxxxx` 的形式存在的，有的是跟随请求体传输的。

**根本原因**：系统只支持一种 API 密钥传输方式，无法适应不同 AI 引擎的要求。

**解决方案**：

- 在 `AIEngineConfig` 接口中添加了 `api_key_transmission` 字段，支持 `header`（Authorization: Bearer）和 `body` 两种传输方式
- 在默认配置中为该字段设置了默认值 `body`
- 在 AI 引擎设置部分添加了 API 密钥传输方式的选择选项
- 更新了 `handleSave` 函数，确保在保存设置时也保存 API 密钥传输方式
- 更新了 `handleSaveEngine` 函数，确保在添加或编辑引擎时也包含 API 密钥传输方式
- 更新了 `useEffect` 钩子，确保在配置变化时，将 `api_key_transmission` 字段也设置到表单中

**预防措施**：

- 设计系统时应考虑到不同 API 的差异，提供灵活的配置选项
- 确保所有配置选项都能正确保存和加载

### 12. AI 翻译和润色功能调用失败

**问题**：虽然连通性测试成功但是 AI 翻译和 AI 润色功能调用失败，`WorldBookManager.tsx:1341 POST https://ai-api.crec.cn/v1/chat/completions net::ERR_CONNECTION_CLOSED`

**根本原因**：前端直接发送请求导致的 CORS 问题，以及请求超时设置过短。

**解决方案**：

- 创建了 `AI 请求处理器`，在主进程中处理 AI 请求，避免前端直接发送请求导致的 CORS 问题
- 实现了 `ai:request` IPC 接口，供前端调用
- 将所有 AI 相关功能的代码修改为使用 IPC 接口发送请求
- 将超时时间从 30 秒增加到 120 秒，给本地模型足够的处理时间

**预防措施**：

- 对于需要跨域的 API 请求，使用主进程处理，避免 CORS 问题
- 为本地模型设置合理的超时时间，考虑到本地模型的处理速度

### 13. API 密钥 Bearer 前缀重复问题

**问题**：API 密钥传输方式: header，AI 引擎错误返回: `{ "error": { "message": "连接失败，请检查 API 地址和密钥", "type": "connection_error", "code": 401 } }`

**根本原因**：系统在添加 Bearer 前缀时没有检查 API 密钥本身是否已经包含 Bearer 前缀，导致出现 "Bearer Bearer sk-xxx" 这样的重复情况。

**解决方案**：

- 修改了所有添加 Bearer 前缀的代码，添加了检查逻辑
- 当 API 密钥已经包含 "Bearer " 前缀时，直接使用该密钥
- 当 API 密钥不包含 "Bearer " 前缀时，添加前缀后使用

**预防措施**：

- 在处理 API 密钥时，应检查其格式，避免重复添加前缀
- 提供清晰的用户提示，说明 API 密钥的正确格式

### 14. 模型名称使用错误

- 修改了 `handlePolish` 函数，使其使用动态的 AI 引擎配置
- 在所有 AI 相关功能中，都使用了动态获取的配置参数，而不是硬编码的配置

**预防措施**：

- 避免在代码中硬编码配置信息，应使用动态获取的配置
- 确保所有 AI 相关功能都使用相同的配置获取逻辑

### 15. 本地模型使用时的超时问题

**问题**：当切换到本地模型再进行润色时报了超时错误，本地 AI 服务器是连通的，不是 API key 的问题，本地没有设置 API 秘钥。

- 代码维护困难：当需要修改默认值时，需要在多个地方进行修改

- 提交信息格式：`[类型] 描述`
- 类型包括：feat（新功能）、fix（修复）、docs（文档）、style（样式）、refactor（重构）、test（测试）、chore（构建/依赖）
- 描述要简洁明了，说明本次提交的内容

#### 代码审查
- 所有代码变更都需要经过代码审查
- 审查内容包括：代码质量、功能正确性、安全性、性能
- 至少需要 1 个审查者批准才能合并

## 前端样式风格规范

### 色彩方案
- 主色：#1890ff（Ant Design 蓝色）
- 辅助色：#52c41a（绿色）、#faad14（黄色）、#f5222d（红色）
- 中性色：#f0f0f0（浅灰）、#e8e8e8（中灰）、#8c8c8c（深灰）
- 背景色：#ffffff（白色）
- 文本色：#333333（主文本）、#666666（次要文本）、#999999（辅助文本）

### 组件设计规范
- 使用 Ant Design 组件库
- 保持组件风格一致
- 组件尺寸统一
- 间距规范：8px、16px、24px、32px
- 卡片设计：使用 Ant Design 的 Card 组件，保持统一的阴影和圆角
- 按钮设计：使用 Ant Design 的 Button 组件，根据功能使用不同的类型（primary、default、dashed、link）

#### 记忆插件UI组件规范
1. **树形结构组件**：
   - 使用 Ant Design 的 Tree 组件展示聊天记录
   - 支持多选和全选功能
   - 显示关联状态标识（已关联/未关联）
   - 展开/折叠状态保持一致

2. **表格管理组件**：
   - 使用 Ant Design 的 Table 组件展示 Excel 数据
   - 支持列宽自适应和手动调整
   - 提供分页、排序、筛选功能
   - 单元格编辑时提供即时反馈

3. **模板管理组件**：
   - 使用 Ant Design 的 Tabs 组件实现多页签
   - 每个页签对应一个功能表格
   - 页签切换时保持状态一致性
   - 提供版本控制和历史记录查看

4. **进度显示组件**：
   - 使用 Ant Design 的 Progress 组件显示处理进度
   - 长时间操作显示进度条和百分比
   - 提供取消操作功能
   - 完成后显示成功/失败状态

### 布局原则
- 响应式设计：使用 Ant Design 的 Grid 组件，确保在不同屏幕尺寸下的良好显示
- 网格布局：使用 24 列网格系统，合理分配空间
- 层次分明：使用卡片、阴影、边框等元素创建视觉层次
- 视觉平衡：保持页面元素的平衡分布，避免过于拥挤或稀疏

#### 记忆插件页面布局规范
1. **聊天记录管理页面**：
   - 左侧：树形结构展示聊天记录，占比 30%
   - 右侧：聊天记录详情和操作区域，占比 70%
   - 底部：操作按钮区域（关联模板、整理表格等）
   - 使用 SplitPane 或 Flexbox 实现可调整的左右分栏

2. **模板管理页面**：
   - 顶部：模板列表和操作按钮
   - 中间：多页签表格编辑区域
   - 底部：版本历史和保存按钮
   - 固定高度的表格区域，支持滚动

### 字体规范
- 主字体：-apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, 'Helvetica Neue', Arial, sans-serif
- 字体大小：12px、14px、16px、18px、20px、24px
- 字体粗细：400（正常）、500（中等）、600（半粗）、700（粗）
- 行高：1.4（正常）、1.5（宽松）

### 样式命名规则
- 使用 BEM（Block Element Modifier）命名规范
- 组件样式文件与组件文件同名，使用 `.css` 或 `.module.css` 扩展名
- 避免使用内联样式，优先使用 CSS 类
- 使用 CSS 变量定义主题色和间距
- 暗色模式样式使用 `.dark` 类前缀

### 交互体验要求
- 反馈及时：操作后立即显示反馈（如 loading 状态、成功/失败提示）
- 动画流畅：使用适当的动画效果，提升用户体验
- 操作便捷：减少操作步骤，提供快捷键和批量操作
- 错误处理：提供清晰的错误提示和解决方案
- 响应迅速：确保界面响应时间不超过 200ms

#### 记忆插件交互体验规范
1. **关联模板流程**：
   - 选中聊天记录后动态显示"关联模板"按钮
   - 点击按钮弹出模板选择模态框
   - 模板列表显示模板名称、描述、创建时间
   - 确认关联后显示成功提示和关联状态标识

2. **表格整理流程**：
   - 仅在已关联模板时"整理表格"按钮可点击
   - 点击后显示进度条和当前处理步骤
   - 提供取消操作功能
   - 完成后显示整理结果预览，允许用户确认

3. **模板编辑体验**：
   - 页签切换时自动保存当前页签内容
   - 提供撤销/重做功能
   - 表头编辑支持拖拽排序
   - 实时预览模板效果

- 大屏（≥1200px）：多列布局，充分利用空间
- 中屏（768px-1199px）：适当调整布局，保持良好显示

- 触摸设备：增大点击区域，支持触摸操作

#### 记忆插件响应式适配
1. **大屏（≥1200px）**：
   - 左右分栏布局，左侧30%，右侧70%

   - 充足的操作空间

2. **中屏（768px-1199px）**：
   - 保持左右分栏，但调整比例
   - 表格列适当减少
